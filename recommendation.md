# Production Deployment Recommendations

The current system is architected for single-machine operation with SQLite, which is ideal for demonstrating the core crawling and indexing algorithms. To deploy this into a production environment serving real workloads, the following changes would be prioritized.

**Database and storage.** SQLite with WAL mode works well for single-process read/write concurrency, but a production crawler would benefit from PostgreSQL (or a similar RDBMS) for higher write throughput under heavy concurrent worker loads, proper connection pooling, and the ability to run the API and crawler processes independently. The inverted index could eventually be migrated to a dedicated search engine like Elasticsearch or Meilisearch for sub-millisecond full-text queries at scale, though the current SQL-based approach is surprisingly capable for millions of terms.

**Scaling the crawler.** The `worker_threads` pool handles concurrency well on a single machine, but production scale would benefit from decoupling the crawl orchestrator from the API server. A message queue (Redis Streams, RabbitMQ, or SQS) between the job scheduler and a fleet of stateless crawler workers would allow horizontal scaling, independent failure recovery, and cleaner back pressure semantics. Each worker would pull URLs from the queue, fetch and parse pages, and push results back — enabling the system to scale across multiple machines when needed. Additionally, `robots.txt` compliance should be implemented (fetching and caching each domain's robots.txt before crawling), along with configurable per-domain politeness delays beyond the current fixed 200ms.

**Observability and operations.** Structured JSON logging (via Pino or Winston) piped to a log aggregator, plus Prometheus metrics exported from the Nest.js process (crawl rate, queue depth, error rate, worker utilization) and visualized in Grafana, would provide the operational visibility needed for a production system. The application should be containerized with Docker (multi-stage builds for minimal image size) and orchestrated via Docker Compose for local/staging or Kubernetes for production, with health check endpoints and graceful shutdown handling already partially in place via Nest lifecycle hooks.
